# AICA Execution & Integration Flow 
# AICA 執行與整合流程

> Integrating AICA is not merely running a linter; it is the enforcement of an insurmountable formal boundary within your development lifecycle.
> 導入 AICA 並非單純執行靜態掃描，而是在開發生命週期中強制植入一道不可逾越的形式化邊界。

---

## Phase 1: Establish Static Interface Contracts / 確立靜態介面契約

[EN]
System security cannot rely on black-box guessing; it requires explicit declaration. The first step is to establish a physical directory for policies within the target repository and define the boundary rules.

1. **Create Directory:** Create `.github/aica-rules/` at the root of your target repository.
2. **Inject Policies:** Copy the formal JSON contracts (e.g., `threadlocal-boundary.json`, `time-coupling.json`) into this directory.
3. **Customize Boundaries:** Architects must mutate the `illegal_invocations` blacklist based on the specific topology of their system (e.g., Spring Boot or Node.js BFF), establishing an exclusive "Zero Trust Baseline" for the repository.

[ZH]
系統安全性不能依賴黑盒猜測，必須透過顯式宣告。第一步是在目標專案中建立存放政策的實體目錄，並寫入邊界規則。

1. **建立目錄：** 在目標專案根目錄下建立 `.github/aica-rules/`。
2. **注入政策：** 將形式化 JSON 契約（如 `threadlocal-boundary.json`、`time-coupling.json`）複製到該目錄下。
3. **客製化邊界：** 架構師需依據自身系統（如 Spring Boot 或 Node.js BFF）的拓樸特性，增刪 `illegal_invocations` 黑名單，確立該專案專屬的「零信任基準線」。

---

## Phase 2: Deploy the Gatekeeper / 掛載神經-符號守門員

[EN]
Compile the AICA engine and integrate it into the Continuous Integration (CI) pipeline, making it a mandatory gate before any code merge.

1. **Compile Core Engine:** Build the project to bundle all TypeScript logic and AST dependencies into a single, deterministic `dist/index.js` execution environment.
2. **Configure Workflow:** Create `.github/workflows/aica-gatekeeper.yml` in the target repository. Properly configure the `rules-path` parameter to explicitly inject the static contracts defined in Phase 1 into the GitHub Action.

[ZH]
將 AICA 引擎編譯並整合至持續整合（CI）管線，使其成為每次程式碼合併前的必經閘口。

1. **編譯核心引擎：** 構建專案，將所有 TypeScript 邏輯與 AST 依賴打包進單一的 `dist/index.js`，確保執行環境的絕對收斂與決定性。
2. **配置工作流：** 在目標專案中建立 `.github/workflows/aica-gatekeeper.yml`，並正確配置 `rules-path` 參數，將階段一的靜態契約顯式注入給 GitHub Action。

## Phase 3: Trigger & Mechanistic Interception / 觸發與底層攔截

[EN]
AICA executes passively and absolutely. Its enforcement power triggers the moment developers rely on LLMs to generate code.

1. **AI-Assisted Development:** A developer utilizes AI to generate business logic (e.g., the AI hallucinates a shortcut using `ThreadLocal` to access user identity).
2. **Initiate Pull Request:** The developer pushes the code containing Implicit Coupling and opens a PR.
3. **Trigger AST Slicing:** GitHub Actions automatically invokes AICA. `ast-slicer.ts` scans the PR diff, parses the code into an Abstract Syntax Tree, and matches it strictly against the contracts in `.github/aica-rules/`.
4. **Status Block:** Upon detecting implicit invocations, the CI status is immediately set to `Failure`. The native GitHub Merge button is locked, physically preventing the code from polluting the main branch.

[ZH]
AICA 的執行是被動且絕對的。它將在開發者依賴 LLM 生成程式碼時展現其強制力。

1. **AI 輔助開發：** 開發者使用 AI 快速生成業務邏輯（例如，AI 為了走捷徑，偷偷使用了 `ThreadLocal` 來獲取使用者身分）。
2. **發起合併請求 (PR)：** 開發者將包含隱性耦合的程式碼推送至遠端，並發起 Pull Request。
3. **觸發切片 (AST Slicing)：** GitHub Actions 自動啟動 AICA。`ast-slicer.ts` 掃描 PR 變更，將程式碼解析為語法樹，並精確比對 `.github/aica-rules/` 中的契約。
4. **狀態阻斷 (Status Block)：** 發現隱性呼叫後，CI 狀態立即設為 `Failure`。GitHub 原生的 Merge 按鈕將被強制鎖死，阻止該程式碼流入主分支。

## Phase 4: Closed-Loop Refactoring & Extensionalization / 閉環重構與外延化

[EN]
This is the core engineering value of the framework: transforming a simple "error log" into a closed-loop that explicitly guides the AI to self-correct.

1. **Receive Anti-Corruption Prompt:** Below the violating line of code in the PR, the developer will find an automated comment generated by AICA. This contains the refactoring directive and structured IR calculated by the Extensionalization Operator.
2. **Constrain the LLM:** The developer does not need to rewrite the code manually. They simply copy the prompt and feed it back to their AI assistant. This prompt acts as an absolute constraint, forcing the LLM to abandon implicit states and "lift" the variables into explicit function parameters.
3. **Verify Convergence:** The developer pushes the AI-refactored code. AICA rescans to confirm all implicit dependencies have been transformed into Extensional Semantics. The CI status switches to `Success`, restoring the bisimulation relationship and permitting the merge.

[ZH]
這是整個框架最具工程價值的一環，將單純的「報錯」轉化為「引導 AI 修正」的閉環。

1. **接收防腐提示詞：** 開發者在 PR 的違規程式碼行數下方，會看到 AICA 自動留下的 Comment，內容包含了由外延化投影算子計算出的重構指令與結構化 IR。
2. **約束 LLM：** 開發者無需手動修改程式碼。他們只需複製該段 Prompt，貼回給 AI 助手。這段 Prompt 帶有絕對的約束力，會強制 LLM 放棄隱性狀態，將變數「提升」為顯式的函數參數。
3. **驗證收斂：** 開發者將 LLM 修正後的程式碼再次 Push。AICA 重新掃描，確認所有的隱性依賴皆已轉化為外延語意（Extensional Semantics）。CI 狀態轉為 `Success`，系統恢復互模擬關係，允許合併。
